## P0: 压缩触发顺序（trigger_count=15 < MAX_HISTORY=20）

**测试目标**：验证对话达到 15 条消息时自动触发压缩，而不是等到 20 条被硬截断。

**测试方法**：连续发送消息，观察第 15 条后是否触发压缩。

<p align="center"><font face="黑体" size=4>P0 压缩触发测试用例</font></p>
<div class="center">

| 序号 | 发送的消息 | 预期行为 |
|------|-----------|---------|
| 1 | `你好` | 正常回复，无压缩 |
| 2 | `介绍下Python` | 正常回复，无压缩 |
| 3 | `什么是装饰器` | 正常回复，无压缩 |
| 4 | `解释下闭包` | 正常回复，无压缩 |
| 5 | `列表推导式怎么写` | 正常回复，无压缩 |
| 6 | `字典有哪些常用方法` | 正常回复，无压缩 |
| 7 | `什么是生成器` | 正常回复，无压缩 |
| 8 | `协程和线程的区别` | 此时消息总数达到 15 条（7轮×2 + 第8轮user=15），**后台日志应出现 `[compress]` 相关输出** |

</div>

> 每轮对话产生 user + assistant 两条记录，第 8 轮的 user 消息使总数达到 15，触发自动压缩。压缩后对话摘要被存入 `sessions/` 文件的 `compressed_context` 字段。

**验证方式**：

```bash
# 查看后端日志中的压缩触发
grep "compress" backend.log

# 直接查看 session 文件，确认 compressed_context 字段存在
cat backend/sessions/<session_id>.json | python -m json.tool | grep compressed_context
```

## P1-b: Prompt Caching 前缀稳定性

**测试目标**：静态组件（SOUL/IDENTITY/USER/AGENTS）始终在动态内容（Memory）之前，保证 DeepSeek 自动 Prefix Caching 命中。

> 此项为内部架构优化，无法通过用户 Prompt 直接观测。验证方式如下：

```python
# 在 backend/ 目录运行，打印 system prompt 组件顺序
import sys; sys.path.insert(0, ".")
from pathlib import Path
from graph.prompt_builder import build_system_prompt

base = Path(".")  # backend/ 目录
prompt = build_system_prompt(base)

# 检查组件顺序
components = ["Soul", "Identity", "User Profile", "Agents Guide", "Long-term Memory"]
positions = {}
for c in components:
    marker = f"<!-- {c} -->"
    pos = prompt.find(marker)
    if pos >= 0:
        positions[c] = pos
        print(f"  {c}: position {pos}")

# 验证：前 4 个组件都在 Memory 之前
memory_pos = positions.get("Long-term Memory", float("inf"))
for c in ["Soul", "Identity", "User Profile", "Agents Guide"]:
    assert positions.get(c, float("inf")) < memory_pos, f"{c} should be before Memory!"
print("\n✅ Prefix Caching 前缀顺序验证通过")
```

## P2-a: Token Budget Awareness（两级防线）

**测试目标**：对话历史接近 60K tokens 时发出警告，超过 80K tokens 时强制截断。

> 此测试需要大量消息，建议通过脚本向 session 文件注入长消息来模拟。

### 数据准备脚本

```python
# 在 backend/ 目录下运行，向指定 session 注入长消息模拟 Token Budget 触发
import json
from pathlib import Path

session_file = Path("sessions/token-budget-test.json")
messages = []
# 每条消息约 4000 tokens（6000 中文字符 × 1/1.5）
for i in range(20):
    role = "user" if i % 2 == 0 else "assistant"
    messages.append({
        "role": role,
        "content": f"第{i}轮：" + "这是一段很长的测试内容，用于验证Token预算感知功能。" * 300
    })

data = {
    "title": "Token Budget Test",
    "created_at": 0,
    "updated_at": 0,
    "messages": messages
}
session_file.write_text(json.dumps(data, ensure_ascii=False), encoding="utf-8")
print(f"已注入 {len(messages)} 条消息，预计约 {len(messages) * 4000} tokens")
```

### 测试用例

<p align="center"><font face="黑体" size=4>Token Budget 两级防线测试</font></p>
<div class="center">

| 测试 Prompt | 预期行为 |
|------------|---------|
| 在注入长消息的 session 中发送：`继续之前的讨论` | 后端日志出现 WARNING：`Context exceeds critical threshold (xxxxx > 80000 tokens), truncated to N messages`，对话仍正常回复（不崩溃） |

</div>

**验证方式**：

```bash
# 查看后端日志中的 Token Budget 警告
grep -E "Context (approaching|exceeds)" backend.log
```

## P2-b: 压缩摘要质量验证

**测试目标**：LLM 生成的摘要过短或为空时被拒绝，不执行压缩。

> 正常情况下 DeepSeek 不会生成空摘要，此项主要通过单元测试覆盖。手动测试可验证正常摘要被接受。

<p align="center"><font face="黑体" size=4>压缩摘要质量验证测试</font></p>
<div class="center">

| 测试场景 | 方法 | 预期行为 |
|---------|------|---------|
| 正常压缩 | 触发 P0 测试（15 条消息），让系统自动压缩 | 后端日志**不出现** `WARNING: summary too short` 或 `summary is empty`，压缩正常完成 |
| 手动触发 | `POST /api/sessions/{session_id}/compress` | 返回 `{"archived_count": N, "remaining_count": M}`，不返回 500 错误 |

</div>

**验证方式**：

```bash
# 手动触发压缩 API
curl -X POST http://localhost:8000/api/sessions/<session_id>/compress

# 检查是否有质量门控拦截
grep "WARNING: summary" backend.log
```

## P3: Dynamic Tool Loading（动态工具加载）

**测试目标**：不同类型的消息只加载相关工具子集，减少无关工具对 LLM 的干扰。

### 核心测试用例

<p align="center"><font face="黑体" size=4>Dynamic Tool Loading 分类检测测试</font></p>
<div class="center">

| 测试 Prompt | 预期触发分类 | 预期加载的工具 | 不应加载的工具 |
|------------|------------|--------------|--------------|
| `你好，聊聊天` | 无（fallback 全量） | 全部 13 个工具 | — |
| `帮我搜索知识库中关于 FastAPI 的内容` | `knowledge` | core(3) + `search_knowledge_base` + `fetch_url` | `python_repl`, `execute_skill` |
| `执行技能 get_weather` | `skill` | core(3) + `execute_skill` + `create_skill_version` | `python_repl`, `search_knowledge_base` |
| `写一段python代码计算斐波那契数列` | `code_exec` | core(3) + `python_repl` | `execute_skill`, `search_knowledge_base` |
| `请记住我喜欢简洁的回复风格` | `memory`（仅 mem0 模式） | mem0: core(3) + 8个memory工具; markdown: fallback全量 | — |
| `搜索知识库，然后用python代码分析结果` | `knowledge` + `code_exec` | core(3) + `search_knowledge_base` + `fetch_url` + `python_repl` | `execute_skill` |
| `帮我查找这个链接 http://example.com 的内容` | `knowledge` | core(3) + `search_knowledge_base` + `fetch_url` | `python_repl` |
| `运行代码测试一下` | `code_exec` | core(3) + `python_repl` | `execute_skill` |
| `please run this skill for me` | `skill`（非 code_exec） | core(3) + `execute_skill` + `create_skill_version` | `python_repl`（"run" 单独不触发 code_exec） |

</div>

### 验证方式

```bash
# 后端日志会输出每次请求的动态加载信息
grep "Dynamic tool loading" backend.log

# 正常分类检测示例：
# INFO - Dynamic tool loading: categories={'knowledge'}, message_preview=帮我搜索知识库中关于FastAPI的内容

# 纯聊天 fallback 示例：
# INFO - Dynamic tool loading: no category matched, fallback to full tools. message_preview=你好，聊聊天

# 工具加载详情（来自 tools/__init__.py）：
grep "Dynamic load" backend.log
# [tools] Dynamic load: 5 tools from categories=['core', 'knowledge'] (safe=3, moderate=0, dangerous=1)
```
