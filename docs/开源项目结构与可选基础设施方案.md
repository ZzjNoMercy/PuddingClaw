## 1. 发行原则

PuddingClaw 开源时应同时满足两类用户：

- 想快速体验的人：只启动 PostgreSQL、后端和前端，使用本地 Markdown 检索与轻量 PDF parser。
- 想运行完整 RAG 的人：按需增加 Milvus 和 MinerU，不修改业务代码。

因此依赖分成三层：

| 层级 | 组件 | 默认 |
| --- | --- | --- |
| Core | frontend、backend-api、worker、PostgreSQL、本地 Artifact 存储 | 必需 |
| Vector | LlamaIndex Milvus adapter、Milvus、etcd、Milvus 内部 MinIO | 可选 |
| Rich parsing | MinerU 独立服务/GPU 镜像 | 可选 |

PostgreSQL 不是“高级模式”：它是会话、知识库、任务和权限的正常数据库。Milvus 不替代 PostgreSQL，MinerU 也不应嵌入后端进程。

## 3. PostgreSQL 设计

### 3.1 技术选择

- SQLAlchemy 2.x async：ORM、事务与连接池。
- `asyncpg`：FastAPI/worker 的异步 PostgreSQL driver。
- Alembic：schema migration；应用启动时只检查版本，不自动偷偷改表。
- PostgreSQL 原生 UUID、`jsonb`、约束和索引。

### 3.2 数据职责

第一阶段新增知识库表，第二阶段再把现有 JSON 会话平滑迁入 PostgreSQL：

```text
users
sessions
messages

knowledge_bases
knowledge_base_members
documents
document_revisions
parse_runs
artifacts
publications
index_runs
ingestion_jobs
outbox_events
```

迁移策略：

1. 新知识库功能只写 PostgreSQL。
2. 会话先保持 JSON 读写，增加一次性 import command。
3. 双读校验一段时间后切 PostgreSQL 为主。
4. JSON 导出保留为备份/可移植格式，不再作为并发数据库。

### 3.3 配置

```env
DATABASE_URL=postgresql+asyncpg://puddingclaw:puddingclaw@postgres:5432/puddingclaw
DATABASE_POOL_SIZE=10
DATABASE_MAX_OVERFLOW=20
DATABASE_POOL_RECYCLE_SECONDS=1800
```

密码只放 `.env`/secret manager。仓库只提交 `.env.example`，并提供随机密码提示；生产环境不使用示例密码。

## 4. 可选能力的代码接口

### 4.1 Parser

```python
class DocumentParser(Protocol):
    name: str
    async def health(self) -> CapabilityStatus: ...
    async def parse(self, source: SourceDocument) -> ArtifactSet: ...
```

实现：

- `PyPdfParser`：Core，基础 PDF 文本 fallback。
- `PyMuPdfParser`：推荐 Core extra，文本 + 图片。
- `MinerUHttpParser`：只包含轻量 HTTP client；真正 MinerU 位于独立服务。

路由配置：

```env
KNOWLEDGE_PARSER=auto              # auto|pymupdf|pypdf|mineru
MINERU_URL=                        # 空值表示未启用
MINERU_REQUIRED=false             # true 时启动健康检查失败则 API fail fast
```

`auto` 只选择当前健康且允许的数据边界内的 provider。没有 MinerU 时应正常降级，而不是导致整个应用无法启动。

### 4.2 Vector store

```python
class KnowledgeVectorStore(Protocol):
    async def health(self) -> CapabilityStatus: ...
    async def upsert_text(...): ...
    async def upsert_images(...): ...
    async def search(...): ...
    async def delete_document(...): ...
```

实现：

- `DisabledVectorStore`：Core 默认，明确返回 capability disabled。
- `MilvusVectorStoreAdapter`：安装 Milvus extra 且配置 URI 后启用。
- 后续可增加 pgvector adapter，但不应让 domain/application 层依赖具体实现。

配置：

```env
KNOWLEDGE_VECTOR_BACKEND=disabled  # disabled|milvus
MILVUS_URI=http://milvus:19530
MILVUS_TOKEN=
MILVUS_DATABASE=default
```

当用户选择 `publication_mode=indexed`，但 vector backend 为 disabled 时，API 应返回可操作的 capability error，不能悄悄退化为 local 后还标记索引成功。

## 5. Python 依赖拆分

目标 `pyproject.toml`：

```toml
[project]
dependencies = [
  "fastapi",
  "sqlalchemy[asyncio]",
  "asyncpg",
  "alembic",
  "pydantic-settings",
  "pypdf",
]

[project.optional-dependencies]
# PDF 富解析（base 包，GPU 后端在 Docker 内独立安装）
mineru = ["mineru>=3.0"]

# Milvus 向量存储（llama-index-vector-stores-milvus 当前要求 pymilvus<3）
milvus = [
  "pymilvus==2.6.15",
  "llama-index-vector-stores-milvus==1.1.0",
]

dev = ["pytest", "pytest-asyncio", "ruff", "mypy"]
```

具体版本由 `uv.lock` 锁定，并通过 CI/Dependabot 或 Renovate 更新。MinerU 的完整 Python/CUDA 依赖不进入 backend extra，它属于 `mineru/` 镜像或独立脚本；后端只需 HTTP client 调用其 API。

## 6. Compose 拆分

### 6.1 Core `compose.yaml`

只包含：

- `postgres`
- `backend`
- `worker`
- `frontend`

Core 必须能独立通过健康检查，并支持本地 Markdown 检索和 pypdf/PyMuPDF 解析。

### 6.2 Milvus overlay

`deploy/compose.milvus.yaml` 包含 Milvus standalone 所需服务和持久卷，并给 backend/worker 注入：

```yaml
services:
  backend:
    environment:
      KNOWLEDGE_VECTOR_BACKEND: milvus
      MILVUS_URI: http://milvus:19530
  worker:
    environment:
      KNOWLEDGE_VECTOR_BACKEND: milvus
      MILVUS_URI: http://milvus:19530
```

Milvus 自身使用的 MinIO 只属于 Milvus 内部存储；不要默认把应用 Artifact 也耦合到同一个 bucket/凭据。

### 6.3 MinerU overlay

`deploy/compose.mineru.yaml` 添加 `mineru`，并只给 backend/worker 注入 `MINERU_URL`。开源模板禁止提交本机绝对模型路径，例如当前 Compose 中的 `/Users/pet/models`；应改用命名卷或用户通过环境变量传入的路径。

CPU/GPU 差异建议分成两个 overlay 或 image tag：

```text
deploy/compose.mineru.yaml
deploy/compose.mineru-gpu.yaml
```

GPU overlay 只覆盖 device reservation 和 image，不复制整份配置。

### 6.4 启动矩阵

```bash
# Core
docker compose up -d

# Core + Milvus
docker compose -f compose.yaml -f deploy/compose.milvus.yaml up -d

# Core + MinerU
docker compose -f compose.yaml -f deploy/compose.mineru.yaml up -d

# Full
docker compose -f compose.yaml \
  -f deploy/compose.milvus.yaml \
  -f deploy/compose.mineru.yaml up -d
```

可以额外提供 Makefile 别名：`make up`、`make up-vector`、`make up-mineru`、`make up-full`。README 以这些稳定命令为主，降低 Compose 细节对使用者的干扰。

## 7. Capability 发现与健康检查

增加统一接口：

```text
GET /api/capabilities
```

示例：

```json
{
  "database": {"enabled": true, "healthy": true, "provider": "postgresql"},
  "knowledge": {"enabled": true},
  "vector": {"enabled": false, "provider": "disabled"},
  "parsers": {
    "pypdf": {"enabled": true, "healthy": true},
    "mineru": {"enabled": false, "healthy": false}
  }
}
```

前端根据 capability 隐藏或禁用不支持的发布模式，并显示启用说明。后端仍必须校验，不能把安全边界交给前端。

## 8. 开源仓库必须补齐的文件

当前 README 写了 MIT，但根目录尚未发现实际 `LICENSE` 文件。公开前至少补齐：

- `LICENSE`：确认项目代码许可证，不能只在 README 写一句。
- `THIRD_PARTY_NOTICES.md`：列出 MinerU、Milvus、模型权重及其他依赖许可证；服务可选不代表许可证责任消失。
- `CONTRIBUTING.md`：开发环境、迁移、测试、提交规范。
- `SECURITY.md`：漏洞报告渠道、支持版本、文件上传安全说明。
- `CODE_OF_CONDUCT.md`。
- `.github/workflows/ci.yml`：Core 单测不依赖 Milvus/MinerU；可选集成测试单独 job。
- `.github/ISSUE_TEMPLATE/` 与 PR template。
- `.env.example` 和脱敏配置，禁止个人绝对路径、API key、内部代理 URL。

模型权重尤其要单独说明：代码许可证、Docker 镜像许可证与模型许可证是三件不同的事。

## 9. CI 测试矩阵

| Job | PostgreSQL | Milvus | MinerU | 目的 |
| --- | --- | --- | --- | --- |
| core-unit | 否/测试替身 | 否 | 否 | 领域逻辑、工具、安全 |
| postgres-integration | 是 | 否 | 否 | migration、repository、lease/outbox |
| knowledge-local | 是 | 否 | 否 | 上传、pypdf/PyMuPDF、glob/rg |
| milvus-contract | 是 | 是 | 否 | schema、upsert/search/delete |
| mineru-contract | 是 | 否 | mock/可选 nightly | HTTP contract、Artifact manifest |
| full-e2e | 是 | 是 | 可选 nightly/self-hosted | 全链路图文问答 |

普通贡献者执行 Core 和 PostgreSQL 测试即可；GPU/MinerU 测试不能阻塞所有 PR。

## 10. 从当前仓库渐进迁移

### Step 1：数据库先行

- 在当前 `backend/` 内加入 `persistence/` 和 Alembic。
- Core Compose 加 PostgreSQL；实现 `DATABASE_URL`。
- 知识库新表直接使用 PostgreSQL，不新增 SQLite。

### Step 2：应用与 worker 分离

- 保持同一镜像，使用不同 command 启动 API/worker。
- 抽出 `knowledge/domain`、`application`、`infrastructure` 边界。

### Step 3：可选依赖

- `search_knowledge_tool` 只依赖 application service。
- MinerU 改成 HTTP parser adapter。
- Milvus import 放进 adapter 内部，未安装 extra 时提供清晰 capability 状态。

### Step 4：Compose overlay

- 将当前强制 `mineru` 服务移出 Core。
- 删除个人绝对模型目录。
- 增加 Milvus overlay 和启动矩阵测试。

### Step 5：开源包装

- 补齐许可证、贡献、安全、CI、release 和 migration 文档。
- 再决定是否物理迁移为 `apps/ + packages/`，不为了目录漂亮破坏稳定性。

## 11. 关键决策摘要

1. PostgreSQL 从知识库第一版开始就是必需基础设施。
2. Milvus、MinerU 是可选 runtime capability，不是 Core 启动前提。
3. MinerU 作为独立服务发布，后端不安装其重量级依赖。
4. Python extras 控制 adapter，Compose overlays 控制服务，两者必须同时满足才算 capability 可用。
5. Core 在没有 Milvus/MinerU 时必须完整可运行，并给出明确降级能力。
6. README 区分“当前已实现”和“目标架构”，不能把路线图写成现成功能。
