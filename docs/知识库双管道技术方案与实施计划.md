# 知识库双管道技术方案与实施计划

> 方案来源：[`/Users/pet/Code/AI/Agent/llamaindex/llamaindex_courses/进阶 01 图文混排PDF检索.ipynb`](../../../../llamaindex/llamaindex_courses/进阶%2001%20图文混排PDF检索.ipynb)  
> 目标：在 PuddingClaw 中落地支持图文混排 PDF 的多模态 RAG 知识库。

---

## 1. 背景与目标

### 1.1 当前痛点

PuddingClaw 现有的 `MemoryIndexer` 只能处理 `MEMORY.md` 纯文本，存在以下局限：

- 无法处理 PDF、PPT、图片等富文档；
- 无法提取和利用文档中的图片、架构图、表格；
- 多模态信息割裂，检索时只能召回文本，视觉信息丢失。

### 1.2 目标

构建一套**图文双管道**知识库方案：

- **文本管道**：提取 PDF 中的文本、表格、Markdown 结构，向量化后存入文本向量库；
- **图片管道**：提取 PDF 中的图片、图表，向量化后存入图片向量库；
- **统一检索**：查询时同时检索文本和图片，由多模态 LLM（如 Qwen-VL）综合生成答案。

---

## 2. 核心架构：双管道（Text + Image）

```
┌─────────────────────────────────────────────────────────────┐
│                        用户 Query                          │
└──────────────────────┬──────────────────────────────────────┘
                       │
                       ▼
┌─────────────────────────────────────────────────────────────┐
│              Query 向量化（文本 embedding）                  │
└──────────────────────┬──────────────────────────────────────┘
                       │
         ┌─────────────┴─────────────┐
         ▼                           ▼
┌─────────────────────┐   ┌─────────────────────┐
│   文本向量检索       │   │   图片向量检索       │
│   Text Collection   │   │   Image Collection  │
│   (Milvus)          │   │   (Milvus)          │
└─────────┬───────────┘   └─────────┬───────────┘
          │                         │
          ▼                         ▼
┌─────────────────────────────────────────────────────────────┐
│            检索结果融合（TextNode + ImageNode）              │
└──────────────────────┬──────────────────────────────────────┘
                       │
                       ▼
┌─────────────────────────────────────────────────────────────┐
│              多模态 LLM 生成答案（Qwen-VL）                  │
└─────────────────────────────────────────────────────────────┘
```

**关键设计原则**：

- **分流解耦**：文本和图片分别提取、分别索引、分别存储；
- **统一查询**：同一个查询向量同时检索文本库和图片库；
- **视觉推理**：检索到的图片直接送入多模态 LLM 进行理解。

## 4. 详细方案

### 4.1 文档解析层：MinerU

MinerU 采用 **VLM 版面分析 + OCR 文字识别** 的双模型架构：

1. **VLM 版面分析**：识别页面中的标题、正文、图片、表格、公式等区域及阅读顺序；
2. **OCR 兜底**：对扫描件或编码层缺失的区域调用 PaddleOCR；
3. **图片提取**：将图片区域保存为独立文件；
4. **Markdown 重组**：按阅读顺序生成结构化 Markdown。

关键输出文件：

| 文件 | 作用 |
|------|------|
| `{uuid}_content_list.json` | 线性内容列表，含 `text` / `image` 类型及 `img_path` |
| `{uuid}_model.json` | 语义块模型，VLM 分类结果 |
| `layout.json` | 原始 OCR 与布局层级信息 |
| `full.md` | 最终 Markdown |
| `images/` | 提取的图片资源 |

### 4.2 Embedding 层：多模态向量模型

#### 4.2.1 文本 Embedding

使用 `text-embedding-v4`（OpenAI 兼容），可通过 Higress 网关调用：

```python
from llama_index.embeddings.openai import OpenAIEmbedding

embed_model = OpenAIEmbedding(
    model="text-embedding-v4",
    api_key="...",
    api_base="http://localhost:8080/v1"  # 经本机 Higress 网关路由
)
```

#### 4.2.2 图片 / 多模态 Embedding

`qwen3-vl-embedding` 不支持 OpenAI 兼容接口，需直接调用 DashScope 多模态接口：

```python
import dashscope
from llama_index.core.embeddings import MultiModalEmbedding
from llama_index.core.schema import ImageType
from typing import List

class DashScopeMultiModalEmbedding(MultiModalEmbedding):
    def __init__(self, api_key: str, model_name: str = "qwen3-vl-embedding"):
        super().__init__(model_name=model_name)
        dashscope.api_key = api_key

    def _call_api(self, input_data: list) -> List[float]:
        resp = dashscope.MultiModalEmbedding.call(
            model=self.model_name,
            input=input_data
        )
        if resp.status_code == HTTPStatus.OK:
            return resp.output["embeddings"][0]["embedding"]
        return [0.0] * self.dimension

    def _get_image_embedding(self, img_file_path: ImageType) -> List[float]:
        abs_path = Path(img_file_path).absolute()
        return self._call_api([{"image": f"file://{abs_path}"}])

    def _get_text_embedding(self, text: str) -> List[float]:
        return self._call_api([{"text": text}])

    def _get_query_embedding(self, query: str) -> List[float]:
        return self._call_api([{"text": query}])
```

**关键参数**：

- `dimension`：默认 2560，可指定 1024 / 2048 等；
- `enable_fusion=True`：将文本+图片融合为一个向量；
- 双管道场景下建议**独立向量**（`enable_fusion=False`），文本和图片分别入各自的 Collection。

### 4.3 向量存储层：Milvus 双 Collection

```python
from llama_index.vector_stores.milvus import MilvusVectorStore
from llama_index.core import StorageContext

MILVUS_URI = "http://localhost:19530"
DIM = 1024

# 文本库
text_store = MilvusVectorStore(
    uri=MILVUS_URI,
    collection_name="rag_prod_text",
    dim=DIM,
    overwrite=False,
    enable_sparse=False
)

# 图片库
image_store = MilvusVectorStore(
    uri=MILVUS_URI,
    collection_name="rag_prod_image",
    dim=DIM,
    overwrite=False,
    enable_sparse=False
)

# 组合 StorageContext
IMAGE_VECTOR_STORE_KEY = "image"
storage_context = StorageContext.from_defaults(vector_store=text_store)
storage_context.vector_stores[IMAGE_VECTOR_STORE_KEY] = image_store
```

**注意维度一致性**：

- `text-embedding-v4` 默认输出 1536 维；
- `qwen3-vl-embedding` 默认输出 2560 维；
- 若文本和图片使用不同模型，需保证各自 Collection 的 `dim` 与模型输出一致。

本方案建议文本和图片统一使用 `qwen3-vl-embedding`，这样两库维度相同，检索时可用同一查询向量。

### 4.4 索引与检索层：MultiModalVectorStoreIndex

```python
from llama_index.core.indices.multi_modal import MultiModalVectorStoreIndex

index = MultiModalVectorStoreIndex.from_documents(
    documents,                       # MinerU 解析后的 Document + ImageDocument
    storage_context=storage_context,
    embed_model=embed_model,         # 文本 embedding
    image_embed_model=embed_model,   # 图片 embedding
    image_vector_store_key="image",
    show_progress=True
)
```

索引构建完成后，LlamaIndex 会：

- 将 `TextNode` 存入 `rag_prod_text`；
- 将 `ImageNode` 存入 `rag_prod_image`；
- 在 `docstore` 中保存节点关系。

### 4.6 Agent 集成

将多模态检索封装为 LangChain Tool，供 Agent 动态选择：

```python
from langchain.tools import tool

@tool
def search_multimodal_documents(query: str) -> str:
    """多模态检索（同时搜索文本和图片）。"""
    response = multimodal_query_engine.query(query)
    return format_response(response)

@tool
def search_text_documents(query: str) -> str:
    """纯文本检索（速度更快）。"""
    response = text_query_engine.query(query)
    return format_response(response)
```

Agent 工具选择策略：

- 问题涉及“图”“架构图”“示意图”“可视化” → 使用 `search_multimodal_documents`；
- 纯文本概念/定义类问题 → 使用 `search_text_documents`；
- 需要计算 → 使用 `calculator`。

## 6. 关键注意事项

### 6.1 Higress 网关限制

- `qwen3-vl-embedding` **不能**走 Higress 的 OpenAI 兼容路由，会报错：
  ```json
  {"error": "Unsupported model `qwen3-vl-embedding` for OpenAI compatibility mode."}
  ```
- 多模态 embedding 必须在 backend 中直接调用 DashScope 原生接口；
- 只有 chat 模型（`qwen-vl-*`）和文本 embedding（`text-embedding-v4`）适合走 Higress。

### 6.2 维度对齐

- 文本 Collection 和图片 Collection 的 `dim` 必须分别与对应 embedding 模型输出维度一致；
- 如果统一使用 `qwen3-vl-embedding`，两库维度均为 2560（或显式指定相同维度）。

### 6.3 CLIP 模型的 77 token 限制

如果使用 CLIP 类模型做图片 embedding，查询文本不能超过 77 tokens。因此检索时常用 **Small-to-Big** 策略：

- 用简短查询向量检索；
- 再用完整问题交给多模态 LLM 生成答案。

### 6.4 Milvus 连接问题

pymilvus 2.6.x 存在异步事件循环冲突，建议在导入 `MilvusVectorStore` 前设置：

```python
import asyncio
import nest_asyncio

loop = asyncio.new_event_loop()
asyncio.set_event_loop(loop)
nest_asyncio.apply(loop)
```

### 6.5 成本与性能

| 项目 | 成本/性能 | 说明 |
|------|-----------|------|
| MinerU 本地部署 | 一次性 GPU 成本 | 适合高频、隐私场景 |
| MinerU 在线 API | ~$0.01/页 | 适合低频或快速验证 |
| qwen3-vl-embedding | 按 token/图片计费 | 图片比文本贵，注意控制图片数量 |
| qwen-vl-max | 按 token 计费 | 多模态生成比纯文本贵 |

## 9. 待决策事项

1. **Embedding 模型选择**：
   - 方案 A：文本用 `text-embedding-v4`（1536 维），图片用 `qwen3-vl-embedding`（2560 维）—— 双模型，双维度；
   - 方案 B：文本和图片统一用 `qwen3-vl-embedding`（2560 维）—— 单模型，检索更统一。

2. **图片存储方式**：
   - 本地文件系统 + metadata 存路径；
   - 对象存储（MinIO/OSS）+ 存 URL。

3. **是否启用融合向量**：
   - 独立向量：灵活，适合双管道；
   - 融合向量：语义更完整，但实现复杂。

4. **MinerU 部署方式**：
   - 本地 GPU 部署；
   - 在线 API；
   - Docker 容器（项目已有 `mineru/Dockerfile`）。
