## 1. 结论先行

知识库应设计成一个共享摄取层、两个发布目标，而不是两套互不相干的上传和解析流程：

```text
异构文档上传
    │
    ▼
统一文档摄取层
校验 → 哈希去重 → 原件落盘 → 解析器路由 → 规范化 Artifact
    │
    ├──────────────┬─────────────────────────┐
    ▼              ▼                         ▼
本地 Markdown 管道   LlamaIndex 多模态索引管道      两者同时发布
完整 MD + assets     text/image nodes → Milvus      local + indexed
glob/rg 检索          文本/图片双 collection
```

核心决策：

1. 上传、解析、规范化只做一次；“是否入库”是解析完成后的发布策略。
2. 原始文件、Markdown、图片、表格等 Artifact 永远保存在本地，Milvus 只保存向量、检索字段和稳定 Artifact 引用。
3. 本地管道以完整 Markdown 为事实源，但检索时先用安全封装的 glob + ripgrep 找到文档/段落，再按需读取完整 Markdown，避免把几十万字一次塞进模型上下文。
4. 向量管道使用 LlamaIndex 负责编排节点和检索，Milvus 负责持久化；文本和图片使用独立 collection。
5. MinerU 是 PDF 的首选解析器，PyMuPDF/pypdf 是降级路径；DOCX、PPTX、XLSX、Markdown、纯文本通过各自适配器进入同一规范化格式。
6. 摄取必须异步化并具备任务状态、幂等、重试、删除和重建能力；不能在上传 HTTP 请求或 Agent 工具调用中同步完成 OCR/Embedding。
7. 现有 `rag_mode` 是长期记忆的上下文策略，不应继续承担知识库产品的总开关。知识库选择应是会话/请求级的 `retrieval_scope`。

## 2. 范围与非目标

### 2.1 本期目标

- 上传 PDF 及常见异构文档。
- 统一解析为可复用的 Markdown、图片、表格及结构元数据。
- 支持 `local`、`indexed`、`both` 三种发布模式。
- 支持纯文本检索和图文联合检索。
- 检索结果进入现有 `answer_context + sources[]` 引用协议。
- 支持文档状态查看、失败重试、重新解析、重新索引、删除和索引版本迁移。

### 2.2 暂不包含

- 多租户计费、复杂组织权限和文档审批流。
- 对图片、表格进行独立在线编辑。
- 首期即引入完整 Elasticsearch/OpenSearch 集群。
- 让 Agent 自己执行任意 shell `grep`；本地检索必须是参数化、只读的后端能力。

## 3. 当前项目基础与需要替换的部分

### 3.1 可以复用

| 现有能力 | 复用方式 |
| --- | --- |
| MinerU 自带 `mineru-api` | 通过 `scripts/setup-mineru.py` 部署，解析器 worker 调用其 API 并规范化 Artifact |
| `backend/tools/mineru_tool.py` | 保留为临时单文件问答工具；知识库摄取不直接依赖 Agent 工具 |
| `backend/tools/search_knowledge_tool.py` | 保留工具名和引用 envelope，内部替换为统一检索服务 |
| `backend/graph/citations.py` | 复用 `document_id/chunk_id/page/quote/score` 来源模型 |
| `SourcesPanel` | 增加图片缩略图、文档打开/定位能力即可 |
| Docker Compose | Core 增加 PostgreSQL/worker；Milvus、MinerU 改为可选 overlay |

### 3.2 现状问题

- 当前 `search_knowledge_base` 首次调用时扫描整个 `backend/knowledge` 并同步建本地索引，上传后也没有可靠的增量失效机制。
- 当前 MinerU 服务只返回 Markdown 字符串，临时目录随后被删除，图片和版面 Artifact 无法被多模态索引长期引用。
- `output_files` 返回的是容器临时路径，响应结束后路径已经失效。
- 当前知识库索引和 `MemoryIndexer` 都使用 LlamaIndex 全局 `Settings`，不同 embedding 或并发任务之间容易互相污染。
- 没有文档目录、摄取任务、解析版本、索引版本、发布状态和删除语义。
- 当前描述写着“BM25 + Vector”，实际实现是单一 `VectorStoreIndex.as_query_engine()`，尚未形成真正的混合检索。

## 4. 领域模型：一次摄取，多个发布目标

### 4.1 核心对象

```text
KnowledgeBase
  └─ Document（用户视角的一份文档）
       ├─ DocumentRevision（由 sha256 标识的一版原件）
       ├─ ParseRun（解析器、版本、参数和状态）
       │    └─ ArtifactSet
       │         ├─ full.md
       │         ├─ manifest.json
       │         ├─ pages/*.json
       │         ├─ images/*
       │         └─ tables/*
       └─ Publication
            ├─ local
            └─ indexed
                 └─ IndexRun（embedding/index schema 版本）
```

`Document` 与物理文件分开，便于同名更新、版本回滚和引用稳定。`ParseRun` 与 `IndexRun` 分开，embedding 模型升级时只重建索引，不必重新 OCR。

### 4.2 发布模式

| 模式 | 原件/Artifact | 本地检索 | Milvus | 适用场景 |
| --- | --- | --- | --- | --- |
| `local` | 保存 | 开启 | 不写入 | 临时资料、隐私资料、低成本模式 |
| `indexed` | 保存 | 默认关闭，可用于兜底 | 写入 | 高频语义问答、图文检索 |
| `both` | 保存 | 开启 | 写入 | 需要精确字符串与语义/图片召回 |

即便选择 `indexed`，也不能删除本地 Artifact；图片节点、引用打开、重建索引和审计都依赖它。

## 5. 统一摄取与解析层

### 5.1 上传流程

1. 流式上传到隔离的 staging 文件，不把整个文件读入内存。
2. 校验 MIME、扩展名、文件头、大小、页数上限和压缩包风险。
3. 计算 SHA-256；在同一知识库内命中相同 hash 时复用 `DocumentRevision` 和 Artifact。
4. 原件原子移动到 content-addressed 存储。
5. 创建 `ingestion_job`，立即返回 `202 + job_id`。
6. worker 领取任务，执行解析、规范化、发布和索引。

### 5.2 解析器路由

```text
PDF
 ├─ 默认：MinerU（扫描件、中文、公式、复杂版面、图表）
 ├─ 降级：PyMuPDF（可提取文本 + 原图，速度快）
 └─ 最后降级：pypdf（只保证基础文本，不承诺版面和图片）

DOCX → python-docx/unstructured 适配器
PPTX → python-pptx 适配器（页面文本 + 页面渲染图）
XLSX → openpyxl 适配器（sheet → Markdown/CSV + 元数据）
HTML/MD/TXT → 清洗/规范化适配器
图片 → OCR/VLM 适配器
```

解析器不是静默 fallback。每次尝试都记录 `parser_name`、`parser_version`、耗时、警告和失败原因。最终 Artifact 标记质量等级：

- `rich`：文本、图片、页码、阅读顺序和结构完整。
- `text_only`：仅文本可用。
- `partial`：部分页或资源失败，需要 UI 告警。

### 5.3 MinerU 服务改造

当前 `/parse` 建议拆成内部任务接口或由 worker 调用，并返回规范化结果：

```json
{
  "parser": "mineru",
  "parser_version": "...",
  "quality": "rich",
  "markdown_path": "full.md",
  "assets": [
    {
      "type": "image",
      "relative_path": "images/page_3_fig_1.png",
      "page": 3,
      "caption": "系统总体架构",
      "bbox": [72, 180, 520, 610]
    }
  ],
  "warnings": []
}
```

MinerU 容器只负责生成结果，worker 负责把结果复制到稳定 Artifact 目录。任何写入数据库或 Milvus 的动作都不放进 MinerU 服务。

### 5.4 Artifact 规范

```text
backend/data/knowledge/
├─ originals/{sha256[:2]}/{sha256}/source.pdf
├─ artifacts/{document_id}/{revision_id}/{parse_version}/
│  ├─ full.md
│  ├─ manifest.json
│  ├─ pages/0001.json
│  ├─ images/page_0001_fig_01.png
│  └─ tables/page_0002_table_01.md
└─ local/{knowledge_base_id}/{document_id}/
   ├─ document.md          # 可读发布镜像或安全链接
   ├─ assets/
   └─ metadata.json
```

路径使用内部 UUID，显示文件名仅存元数据，避免重名、路径穿越和 Unicode 兼容问题。`local` 是发布视图，不是另一份解析源。

## 6. 管道 A：本地完整 Markdown + glob/grep

用户所说的“glop grep”按 `glob + grep` 实现，底层优先使用 `ripgrep`，但不直接暴露 shell。

### 6.1 检索分两步

1. `search_local_knowledge`：按知识库范围 glob 文档，使用固定参数的 `rg --json` 搜索，返回命中文档、标题路径、行号和相邻上下文。
2. `read_local_knowledge`：按稳定 `document_id` 读取命中标题段、行区间或完整 Markdown。

完整 Markdown 始终保留且可读取；默认先搜后读是为了控制模型上下文，并非截断源文件。

### 6.2 查询能力

- literal 精确字符串搜索，适合编号、API、错误码、中文短词。
- regex 高级搜索，但设置表达式长度、超时和结果上限。
- 标题过滤、文件名过滤、AND/OR 多词查询。
- 大小写、知识库范围、文档范围过滤。
- 无 `rg` 环境回退到 Python 流式扫描，而不是失败。

### 6.3 返回协议

```json
{
  "answer_context": "命中内容……",
  "sources": [
    {
      "title": "设计文档",
      "uri": "knowledge://kb_123/doc_456",
      "document_id": "doc_456",
      "chunk_id": "lines:120-146",
      "source_type": "knowledge_base",
      "page": 7,
      "quote": "命中原文",
      "metadata": {"backend": "local_rg", "heading": "3.2 架构"}
    }
  ]
}
```

Markdown 中插入不可见或结构化的页锚点（例如 `<!-- page:7 -->`），使 grep 行号能映射回原 PDF 页码。

## 7. 管道 B：LlamaIndex + Milvus 多模态索引

### 7.1 节点构建

规范化层输出三类节点：

- `TextNode`：按 Markdown 标题和语义边界切分；保留父级 section、页码范围和相邻图片 ID。
- `ImageNode`：指向稳定图片路径或对象存储 URI；包含 caption、页码、bbox、邻近文本。
- 可选 `TableNode`：首期作为文本节点写入，元数据标记 `content_type=table`；后续再增加表格专用召回。

推荐 small-to-big：向量化 300–600 token 的叶子块，命中后扩展到父标题段；不要直接把整个 `full.md` 做成一个向量。

每个节点使用确定性 ID：

```text
node_id = sha256(document_revision_id + parse_version + node_type + logical_position)
```

这样任务重试可以 upsert，不会重复插入。

### 7.2 Milvus collection 设计

首期使用每种模态一个共享 collection，通过标量字段隔离知识库，不为每个知识库创建 collection：

```text
kb_text_v1
  id, vector, text, knowledge_base_id, document_id, revision_id,
  node_id, page_start, page_end, heading_path, content_type,
  embedding_model, embedding_version, active, created_at

kb_image_v1
  id, vector, image_uri, caption, knowledge_base_id, document_id,
  revision_id, node_id, page, bbox, linked_text_node_ids,
  embedding_model, embedding_version, active, created_at
```

所有检索必须先应用 `knowledge_base_id`、`active=true` 和权限过滤，再进行向量搜索。删除文档时先标记 catalog 为不可见，再异步删除 Milvus 节点和 Artifact，避免删除窗口继续被召回。

### 7.3 Embedding 策略

定义项目级接口，而不是在业务代码中写死 `DashScopeMultiModalEmbedding`：

```python
class MultiModalEmbeddingProvider:
    text_dimension: int
    image_dimension: int
    def embed_texts(...): ...
    def embed_images(...): ...
    def embed_text_for_image_search(...): ...
```

推荐两种可替换实现：

| 方案 | 文本 collection | 图片 collection | 特点 |
| --- | --- | --- | --- |
| 统一多模态模型 | 同一 provider 的 text vector | 同一 provider 的 image vector | 与 notebook 的 Qwen/DashScope 1024 维思路一致，跨模态一致性好 |
| 双模型 | OpenAI 兼容文本 embedding | CLIP/SigLIP 类图片 embedding | 可独立优化成本；两个 collection 可使用不同维度 |

维度来自 provider 自检，不能把 notebook 的 `1024` 当成常量。启动时校验 collection schema 与当前 provider；不匹配则拒绝写入并要求建立 `v2` collection，不能静默混写。

### 7.4 LlamaIndex 使用边界

- 使用 `MultiModalVectorStoreIndex` 组织 text/image vector store。
- 显式传入 `embed_model` 和 `image_embed_model`，不依赖进程全局 `Settings`。
- 生产代码中 collection 永远 `overwrite=False`；重建索引写新版本 collection，再原子切换 alias/配置。
- Milvus 只作为 vector store 时，本地 catalog 仍是文档真相源；不能依赖 LlamaIndex 内存 docstore 保存业务状态。
- 建索引和查询分别封装为 `KnowledgeIndexer`、`KnowledgeRetriever`，不要把 notebook 的 event loop 修补逻辑带进 FastAPI 主线程。

### 7.5 检索策略

```text
用户问题
  ├─ 查询分类：text / visual / mixed
  ├─ 查询改写：保留完整问题 + 生成短视觉检索 query
  ├─ text dense top-k
  ├─ image dense top-k（visual/mixed 时）
  ├─ 可选本地 rg/BM25 lexical top-k
  ├─ RRF/加权融合 + 去重
  ├─ rerank
  └─ 父段扩展 → answer_context + sources[]
```

参考 notebook 中 CLIP 的短 query 限制，图片检索 query 与最终生成问题必须解耦：短 query 用于召回，原始完整问题用于 VLM/LLM 生成答案。

默认路由建议：

- 定义、条款、数字、原句：文本 + lexical，图片关闭。
- 包含“图、截图、架构图、曲线、页面布局”等意图：文本 + 图片。
- 意图不明确：先文本；低置信或命中节点关联图片时再补图片检索。

多模态生成只把 top 1–3 张真正相关图片交给视觉模型；其余图片仍可作为来源展示，避免成本和上下文失控。

## 8. PostgreSQL Catalog 与任务状态

### 8.1 PostgreSQL 是核心依赖

知识库功能从第一期开始使用 PostgreSQL，不再采用 SQLite 过渡。这样可以一次解决并发写入、任务租约、事务、权限关系、JSON 元数据、全文检索扩展和后续多实例部署问题，避免功能稳定后再做一次高风险迁移。

职责边界：

- PostgreSQL 是系统事实源，保存用户、会话、知识库目录、文档版本、任务、发布状态和引用元数据。
- 本地文件或 S3 兼容对象存储保存原件与解析 Artifact。
- Milvus 仅在启用向量能力时保存向量及检索所需的冗余字段，不保存业务真相。
- MinerU 是可选解析服务，不直接访问业务数据库。

后端采用 SQLAlchemy 2.x async repository + `asyncpg`，数据库迁移使用 Alembic。API、worker 和测试共享 domain model，但分别持有自己的连接池；禁止在业务模块中散落裸 SQL 或全局 session。

最小表：

- `users`
- `sessions`、`messages`（逐步替代当前 JSON 会话文件）
- `knowledge_bases`
- `knowledge_base_members`
- `documents`
- `document_revisions`
- `parse_runs`
- `artifacts`
- `publications`
- `index_runs`
- `ingestion_jobs`
- `outbox_events`

所有业务表使用 UUID 主键、`created_at/updated_at`、必要的唯一约束与外键；动态解析元数据可使用 `jsonb`，但状态、所有者、版本和关联关系必须是正常列，不能把整个对象塞进 JSON。

### 8.2 事务与任务领取

- 上传事务只创建文档记录、revision、job 和 outbox 事件，不在事务内执行 OCR/Embedding。
- worker 使用 `SELECT ... FOR UPDATE SKIP LOCKED` 领取任务，并写入 `lease_owner/lease_expires_at`。
- 每个阶段完成后独立提交，外部调用失败不会长期占用数据库事务。
- publication 和 outbox 在同一事务写入，消费者负责幂等发布到本地 Artifact 或 Milvus。
- 文档删除先在 PostgreSQL 标记不可检索，再异步清理向量和文件。

### 8.3 状态机

```text
UPLOADED
  → QUEUED
  → PARSING
  → NORMALIZING
  → PUBLISHING_LOCAL（按模式可跳过）
  → EMBEDDING（按模式可跳过）
  → INDEXING（按模式可跳过）
  → READY | PARTIAL | FAILED | CANCELLED
```

每个阶段记录进度、开始/结束时间、attempt、错误码和可重试标志。任务以租约方式领取；worker 崩溃后超时任务可重新领取。解析和 embedding 都按 hash/version 缓存。

## 10. Agent、引用与前端集成

### 10.1 Agent 工具

对模型暴露两个清晰工具，而不是让一个工具承担全部策略：

- `search_knowledge(query, knowledge_base_ids?, mode=auto)`：统一检索，默认自动路由。
- `read_knowledge_document(document_id, section?, full=False)`：读取本地完整 Markdown 或指定父段。

如果需要更强的可控性，可在内部保留 `search_local`、`search_text`、`search_multimodal`，但不一定全部暴露给模型。

### 10.2 引用

所有结果继续使用现有结构化 envelope。图片来源补充：

```json
{
  "source_type": "knowledge_base",
  "uri": "knowledge://kb_123/doc_456#asset=img_789",
  "document_id": "doc_456",
  "chunk_id": "img_789",
  "page": 3,
  "quote": "图注或视觉模型的受控描述",
  "metadata": {
    "modality": "image",
    "asset_url": "/knowledge-documents/doc_456/assets/img_789",
    "bbox": [72, 180, 520, 610]
  }
}
```

`quote` 不能伪装成图片原文；视觉模型生成的描述必须标为 `generated_caption=true`。

### 10.3 前端页面

新增 `/knowledge` 工作区：

- 知识库列表与新建。
- 拖拽/批量上传。
- 每个文件选择 `仅本地 / 向量入库 / 两者`，默认继承知识库设置。
- 展示解析、索引进度和降级警告。
- 文档详情查看 Markdown、图片、解析器和索引版本。
- 支持重试、重解析、重索引、改变发布模式和删除。

聊天页在会话设置中选择本轮可检索的知识库；不要把它塞回全局 RAG 开关。来源面板支持图片缩略图和打开对应页/Artifact。

## 11. Docker 与部署拓扑

```text
frontend
   │
backend-api ── PostgreSQL
   │  enqueue
   ▼
ingestion-worker ── mineru
   │       │
   │       └─ shared artifact volume
   ├─ embedding provider
   └─ milvus:19530
          ├─ etcd
          └─ minio
```

建议 volumes：

- `postgres_data`：PostgreSQL 数据目录，核心依赖。
- `knowledge_data`：原件、Artifact 和 local 发布镜像。
- `milvus_data`、`etcd_data`、`minio_data`：Milvus standalone 持久化。

macOS 本地开发时 MinerU 可以沿用 host service；worker 通过配置的 `MINERU_URL` 访问。CPU MinerU 只适合开发验证，批量生产解析应使用 GPU 节点或可替换的远端解析 provider。

开源发行不把所有服务塞进一个强制 Compose 文件。推荐：

```text
compose.yaml                         # frontend/backend/worker/postgres
deploy/compose.milvus.yaml           # 可选 milvus/etcd/minio
deploy/compose.mineru.yaml            # 可选 MinerU 服务
deploy/compose.observability.yaml     # 可选监控
```

用户按需叠加：

```bash
# 核心系统 + PostgreSQL，本地 Markdown / PyMuPDF-pypdf 解析
docker compose up -d

# 增加向量知识库
docker compose -f compose.yaml -f deploy/compose.milvus.yaml up -d

# 增加 MinerU 高质量解析
docker compose -f compose.yaml -f deploy/compose.mineru.yaml up -d

# 全能力
docker compose -f compose.yaml \
  -f deploy/compose.milvus.yaml \
  -f deploy/compose.mineru.yaml up -d
```

详见《开源项目结构与可选基础设施方案》；在真正拆分 Compose 前，当前根目录 `docker-compose.yml` 仍属于既有开发配置，不能在 README 中宣称上述命令已经可用。

## 12. 安全、可靠性与可观测性

### 12.1 安全

- 文件名不参与真实存储路径；所有路径进行 root containment 校验。
- MIME、magic bytes、扩展名三重校验。
- 限制单文件大小、总页数、图片像素、解压后大小和任务并发。
- HTML 清洗，禁止解析产物中的脚本执行。
- 本地 grep 仅接受结构化参数，不拼接 shell。
- 下载/资产接口检查知识库访问权限，不能暴露宿主机绝对路径。
- 将文档文本视为不可信数据，检索上下文与系统指令明确隔离，降低文档 Prompt Injection 风险。

### 12.2 一致性

- 原件和 Artifact 先落盘成功，再更新 catalog 状态。
- Milvus upsert 使用确定性 node ID；重试幂等。
- publication 使用 outbox/待办记录，避免 catalog 标 READY 但向量尚未写完。
- 删除采用 `DELETING → DELETED`，失败可恢复。

### 12.3 指标

- 上传成功率、解析成功/降级率、平均每页解析耗时。
- 队列深度、任务租约超时、重试次数。
- embedding 批次耗时/成本、Milvus upsert 和 query P95。
- text/image/local 各通道召回数、融合后命中率、零结果率。
- 引用覆盖率、点击率、无效 Artifact 引用数。

## 13. 测试策略

建立固定的文档金丝雀集合：

1. 原生中文 PDF。
2. 扫描中文合同。
3. 双栏学术论文、公式、脚注。
4. 图表/架构图密集 PDF。
5. 大表格 PDF。
6. DOCX、PPTX、XLSX、Markdown。
7. 同名文件、重复文件、新版本文件。
8. 恶意文件名、超大图片、损坏 PDF、包含 Prompt Injection 的文档。

测试层级：

- 单元：解析路由、manifest 校验、node ID、权限过滤、grep 参数安全。
- 集成：上传 → Artifact → local 发布；上传 → Milvus text/image → retrieval。
- 契约：MinerU 输出、embedding 维度、Milvus schema、引用 envelope。
- 质量：文字问答、精确字符串、图中信息、跨页信息的 Recall@K/MRR 和引用正确率。
- 故障：worker 中断、MinerU 超时、embedding 限流、Milvus 短暂不可用、重复投递。

## 15. 建议首期默认值

| 项目 | 默认 |
| --- | --- |
| 发布模式 | `both`，可由用户改成 `local` |
| PDF parser | `auto`：MinerU → PyMuPDF → pypdf |
| 文本 chunk | 约 400 tokens，overlap 约 60，优先服从 Markdown 标题边界 |
| text top-k | 8，rerank 后 4 |
| image top-k | 3，送视觉模型最多 2 |
| local grep | 最多 20 个 match，最多 8 个文档，随后按需读取父段/全文 |
| 去重范围 | 同一知识库内 SHA-256 去重 |
| 索引写入 | 增量 upsert，禁止 `overwrite=True` |
| 默认检索模式 | `auto`，文本优先，视觉按意图开启 |

这些是可调基线，不应散落成硬编码；统一放入知识库配置并记录到 `index_run`。

## 16. 实施前仍需确认的产品决策

以下决策不会阻碍后端骨架开发，但应在 Phase 1 前半段定稿：

1. 首期是否仅单用户，还是文档从第一天就需要用户/项目级权限字段。
2. `local` 是否表示“绝不调用任何外部 embedding/VLM 服务”；如果是，解析器和 OCR provider 也必须遵守数据边界。
3. 默认发布模式是 `local`、`indexed` 还是 `both`。
4. 首期除 PDF/MD/TXT 外，DOCX/PPTX/XLSX 的优先顺序。
5. 多模态 embedding 首选当前 notebook 的 DashScope/Qwen 路线，还是部署本地模型。

## 17. 不建议的做法

- 不为 local 和 indexed 各上传、各解析一次。
- 不把 MinerU 的临时图片路径直接写入 Milvus。
- 不在请求线程同步跑 MinerU、切块和 embedding。
- 不用文件名作为 document ID，不用 Milvus 作为业务 catalog。
- 不在生产索引加载逻辑中使用 `overwrite=True`。
- 不把整份超长 Markdown 默认塞进一次工具结果；保留完整源文件，先定位再精读。
- 不只保存向量而丢掉解析 Artifact，否则无法重建、预览和可靠引用。
- 不把知识库产品开关与现有长期记忆 `rag_mode` 混为一谈。
